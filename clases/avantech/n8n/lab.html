1. Introducción y Configuración del Entorno Local

El desarrollo de automatizaciones industriales y flujos de integración de datos a menudo se ejecuta en redes corporativas privadas o subredes de misión crítica aisladas de Internet[span_1](start_span)[span_1](end_span). Este laboratorio proporciona la metodología detallada para desplegar e implementar n8n en un entorno local puro, mitigando problemas de restricciones de proxy y firewalls mediante instalaciones independientes máquina a máquina y servicios de datos locales (MySQL / SQL Server)[span_2](start_span)[span_2](end_span).

Nota de Arquitectura: Al correr localmente, n8n interactúa directamente con los sockets locales de los motores de bases de datos de la máquina[span_3](start_span)[span_3](end_span). Esto elimina la latencia de red y garantiza un entorno de pruebas veloz y 100% seguro para los estudiantes[span_4](start_span)[span_4](end_span).

Instalación local a través de Node.js

Para prescindir de contenedores Docker si el proxy bloquea los registros de imágenes, se recomienda usar Node.js nativo en los terminales de los estudiantes[span_5](start_span)[span_5](end_span):

# 1. Instalar n8n globalmente en la máquina
npm install n8n -g

# 2. Configurar variables de entorno locales
set N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=false
set N8N_DIAGNOSTICS_ENABLED=false

# 3. Levantar la instancia local (localhost:5678)
n8n start

2. Ciclo de Vida de los Datos en Entornos Locales

Cada flujo ejecutado en n8n pasa por un ciclo de vida estrictamente estructurado en memoria interna[span_6](start_span)[span_6](end_span):

Fase Mecanismo Local Descripción del Proceso
1. Inserción / Disparo Cron interno o Webhook interno[span_7](start_span)[span_7](end_span) El planificador del sistema operativo o un endpoint de bucle local (localhost) introduce el JSON inicial a la memoria de ejecución de n8n[span_8](start_span)[span_8](end_span).
2. Extracción de Datos Nodos SQL Server / MySQL[span_9](start_span)[span_9](end_span) Se abren sockets TCP internos a los puertos correspondientes (3306 o 1433) ejecutando transacciones atómicas[span_10](start_span)[span_10](end_span).
3. Transformación Pura Nodo Code (JS)[span_11](start_span)[span_11](end_span) El motor V8 local de Node.js procesa las mutaciones binarias sobre los objetos JSON sin intercambio de paquetes externos[span_12](start_span)[span_12](end_span).
4. Cierre y Escritura Destino en File System o DB[span_13](start_span)[span_13](end_span) Persistencia física de logs o archivos estructurados en los discos locales de la estación de trabajo[span_14](start_span)[span_14](end_span).

3. Casos de Estudio Prácticos (Propuestas de Automatización Local)

A continuación se detallan los tres escenarios diseñados específicamente para interactuar de forma exclusiva con los recursos internos del laboratorio informático[span_15](start_span)[span_15](end_span).

Caso de Estudio 1: Auditoría Automatizada (MySQL a SQL Server)

Este flujo se encarga de migrar datos crudos de facturación o transacciones alojados en una base de datos de desarrollo en MySQL hacia el almacén centralizado local en SQL Server de forma calendarizada[span_16](start_span)[span_16](end_span).

  1. Disparador (Cron Node): Configurado para ejecutarse de forma interna a intervalos fijos (ej. cada 5 minutos en el laboratorio)[span_17](start_span)[span_17](end_span).
  2. Lectura (MySQL Node): Ejecuta una consulta local estructurada: SELECT id, cliente, monto, fecha_registro FROM ventas_temporales WHERE procesado = 0;[span_18](start_span)[span_18](end_span)
  3. Transformación (Code Node): Modifica los campos en formato JSON mediante JavaScript estándar[span_19](start_span)[span_19](end_span): 
    for (let item of $input.all()) {
    item.json.cliente = item.json.cliente.trim().toUpperCase();
    item.json.monto_impuesto = item.json.monto * 0.16;
    item.json.fecha_iso = new Date(item.json.fecha_registro).toISOString();
}
return $input.all();
  4. Escritura (SQL Server Node): Inserta los registros procesados mediante una sentencia de inserción masiva en la tabla estructurada final[span_20](start_span)[span_20](end_span).

Caso de Estudio 2: Sistema Local de Monitoreo de Alertas

Este escenario enseña a los estudiantes a simular un bus de eventos y logs. Una aplicación local del laboratorio envía datos de error mediante peticiones HTTP POST internas hacia el Webhook de n8n[span_21](start_span)[span_21](end_span).

  • Disparador (Webhook Node): Configura una URL interna en modo de escucha local: http://localhost:5678/webhook-test/logs-criticos[span_22](start_span)[span_22](end_span).
  • Lógica Condicional (Switch/If Node): Examina el payload entrante. Si el JSON contiene la propiedad "nivel": "CRITICAL", desvía el flujo por la rama verdadera[span_23](start_span)[span_23](end_span).
  • Acción Local (Execute Command Node): Ejecuta un script en el Shell del sistema operativo (lanzar un archivo batch .bat o script de PowerShell que levante una alerta acústica o ventana emergente)[span_24](start_span)[span_24](end_span).

Caso de Estudio 3: Automatización de Respaldos de Base de Datos

Un problema real en la administración de redes locales es la saturación de los discos duros debido a logs acumulados. Este flujo enseña lógica de mantenimiento[span_25](start_span)[span_25](end_span).

  1. Disparador (Schedule Trigger): Se dispara de manera autónoma fuera del horario de clases[span_26](start_span)[span_26](end_span).
  2. Extracción del Dump (Execute Command Node): Ejecuta la utilidad CLI local para compactar la base de datos sin salir de la máquina[span_27](start_span)[span_27](end_span): 
    mysqldump -u root -pPasswordLocalbd inventario_db > C:\Backups\inventario_respaldo.sql
  3. Lectura de Archivos (Read Binary Files): Captura el archivo binario generado en el disco duro local de la máquina[span_28](start_span)[span_28](end_span).
  4. Escritura (Write Binary File): Copia el archivo binario a una unidad de red compartida asignada internamente en el aula[span_29](start_span)[span_29](end_span).

4. Conclusiones para la Práctica Docente

Implementar estos flujos permite a los alumnos entender que la automatización no depende de herramientas en la nube[span_30](start_span)[span_30](end_span). Los conceptos clave de parsing de datos, estructuración de peticiones, sentencias transaccionales SQL y manejo del sistema de archivos local se consolidan perfectamente bajo esta arquitectura local controlada, aislando al laboratorio de cualquier inestabilidad asociada a los proxies institucionales[span_31](start_span)[span_31](end_span).
